# 日々のタスク設計 — 実施率100%のためのタイムスケジュール

設計原則: **「人間は選んで送信するだけ。考える・書くはAIが済ませておく」**。
迷い・ゼロから書く作業・PCが必要な作業を平日から完全に排除する。全タスクはスマホで完結する。

## 平日のタイムスケジュール（合計 約45分/日）

### 朝の通勤（25分）— 1日で最重要の枠

| 順 | タスク | 時間 | 使うもの |
|---|---|---|---|
| 1 | 【裏】GitHub Issue「📋 デイリー・エンゲージメントキット」の返信下書きをThreadsアプリへコピペ送信 | 10分 | GitHubアプリ + Threadsアプリ |
| 2 | 【裏】Threads 周辺投稿にいいね10〜20件（コメントした相手はフォロー） | 5分 | Threadsアプリ |
| 3 | 【表】同Issueに自動生成されたX下書き3本から1本選び、今朝の実数（フォロワー・views等）を差し込んで投稿 | 5分 | Xアプリ |
| 4 | 【表】X で自分より少し先を行く発信者2〜3人に「中身のある」リプ | 5分 | Xアプリ |

> 順序が重要: 裏（収益エンジン）を先に処理する。表は多少雑でも死なないが、裏のエンゲージは収益に直結する。

### 昼休み（10分・任意）

- 【表】朝のポストへの返信に返す。反応が良ければ引用で深掘りポストを1本追加
- やらなくても「不合格」にしない枠（後述のMVD参照）

### 帰りの通勤（15分）

| 順 | タスク | 時間 |
|---|---|---|
| 1 | 【表】2本目のXポスト（学び型 or 質問型。朝と型を変える） | 5分 |
| 2 | 【裏】14時に自動生成された日次KPIレポートを流し読み（数字を眺めるだけ。判断は週末） | 3分 |
| 3 | 【表】明日のネタメモを1行だけ（レポートで気づいた数字の変化など）。Xの下書き or メモアプリ | 2分 |
| 4 | 【表】Xの通知処理・リプ返し | 5分 |

### 帰宅後

**基本ゼロ。** 帰宅後の作業を前提にした計画は、子どもの発熱1回で崩壊し「自分はダメだ」の失敗ループに入る。
帰宅後にやっていいのは「今日できなかった朝タスクの残り」だけ（新規タスクの追加は禁止）。

## 週末のスケジュール（合計 約3時間）

### 土曜（90分・在宅/PC可）

| タスク | 時間 | 枠 |
|---|---|---|
| ペットカメラYouTube 1本作成・予約投稿（撮り溜め素材の切り出し→サムネはSD/Colabで量産した型を使い回し） | 60分 | 表 |
| note 記事の推敲・公開（下書きはAI生成済みのものを直すだけ。隔週でOK） | 30分 | 表 |

### 日曜（60〜90分・裏の30%枠のコア）

| タスク | 時間 | 枠 |
|---|---|---|
| ASP管理画面 → `data/conversions.csv` エクスポート・コミット（リンク解禁後） | 10分 | 裏 |
| 週次判断: KPIレポート1週間分を見て、ジャンル取捨・リンク差し替え・テンプレ再生成の要否を決める（判断だけ。実行はエンジンがやる） | 20分 | 裏 |
| affiliate-engine 保守（Actions失敗の確認・トークン更新Issue対応など。なければゼロ） | 0〜30分 | 裏 |
| 週次ダッシュボードポスト用の数字整理 → X固定ポスト更新（→ 03） | 15分 | 表 |
| 来週のX下書きの方向性をメモ（自動生成へのフィードバック1〜2行） | 5分 | 表 |

## 実施率100%を担保する4つの仕掛け

### 1. MVD（Minimum Viable Day）— 「最低これだけで合格」を定義する

実施率の敵は「全部できない日に全部やらなくなる」こと。合格ラインを2段階にする:

- **合格（最低ライン）**: 朝のエンゲージキット送信（10分）＋ Xポスト1本（5分）。**15分で合格。**
- **満点**: 上記スケジュール全部
- 発熱・残業・トラブルの日も、トイレ・エレベーター待ちの15分で「合格」は維持できる → 連続記録が途切れない → 失敗恐怖症に効く

### 2. タスクの発生場所を GitHub Issue 1箇所に集約する

「今日何やるんだっけ」と考えた瞬間に実施率は下がる。毎朝6時の自動Issueに
**返信下書き（Threads用）＋ X下書き3本 ＋ 今日の実測数字**が全部届く状態にする（実装は → 04）。
スマホのGitHub通知を開けばその日の全タスクが揃っている。

### 3. 記録は自動に任せる（自分で日報を書かない）

実施の証跡は Threads/X の投稿自体と、エンジンが毎日書き込む `metrics_history.jsonl` に自動で残る。
「記録をつける」というタスクを人間から消す。週末の週次判断で自動レポートを見るだけ。

### 4. if-then ルールを先に決めておく（迷いの排除）

| 状況 | ルール |
|---|---|
| 朝、時間がない | MVDだけやる（エンゲージキット送信＋X 1本） |
| ネタが思いつかない | 自動生成下書きをそのまま投稿してよい（完璧より継続） |
| ポストが全然伸びない日が続く | その場で型を変えない。メモに書いて週末の週次判断で変える |
| 悪意あるリプ・晒し | 反論しない。ブロックして終わり（消耗が最大の敵） |
| 数字が悪くて心が折れそう | 「悪い数字も発信ネタ」。実測を正直に出すのがこのアカウントの商品価値 |
| 週末に時間が取れなかった | YouTube・noteは翌週に繰り越し。**日曜の週次判断（20分）だけは必ずやる** |

## 70/30 配分の実測チェック

上記スケジュールの時間内訳（週合計 約7.5時間）:

| 枠 | 時間/週 | 比率 |
|---|---|---|
| 裏（エンゲージキット送信・いいね・週次判断・保守） | 約2.2h | 約30% |
| 表（X運用・YouTube・note） | 約5.3h | 約70% |

→ 制約どおり。裏が30%で収まるのは自動化済みだからであり、**裏の作業を増やしたくなったら
「それはエンジンに実装できないか」を先に検討する**（人間の時間を増やすのは最終手段）。
